pi 工具执行调用链速查

核心分界：模型决定是否产生 toolCall；本地 agent loop 只负责校验、执行、发事件、把 toolResult 放回上下文。

一句话模型

1. 工具被注册并选中AgentSession 构造 tool registry，active tools 写入 agent.state.tools，system prompt 同步重建。
↓
2. 用户 prompt 进入 sessionAgentSession.prompt() 做命令、模板、鉴权、扩展事件，然后调用 agent.prompt()。
↓
3. LLM 请求带上工具 schemastreamAssistantResponse() 构造 Context，其中 tools: context.tools。
↓
4. 模型流式产出 toolCallprovider stream 发 toolcall_start/delta/end，UI 可先看到参数逐步成形。
↓
5. assistant message 结束后执行工具executeToolCalls() 收集 toolCall，按 parallel/sequential 执行。
↓
6. 单个工具执行找工具 → prepareArguments → validateToolArguments → beforeToolCall → tool.execute() → afterToolCall。
↓
7. 结果变成消息并回到模型上下文发 tool_execution_end，创建 ToolResultMessage，追加到 currentContext.messages，下一轮 LLM 读取结果。

事件顺序速记

无工具

agent_start → turn_start → user message_start/end → assistant message_start/update/end → turn_end → agent_end

有工具

assistant message_end → tool_execution_start → tool_execution_update* → tool_execution_end → toolResult message_start/end → turn_end → next turn

常见误解

  • 不是本地代码主动“选择”工具。 本地代码暴露 active tools；模型根据 prompt、schema、上下文输出 toolCall。
  • 工具结果不是直接展示完就结束。 它会变成 toolResult 消息，进入下一次 LLM 请求。
  • TUI 看到的 tool row 不等于工具已经执行。 message_update 里出现 toolCall 时，UI 可以先画 pending row；真正执行从 tool_execution_start 开始。
  • parallel 模式仍保留消息顺序。 completion event 可按完成顺序出现，但最终 toolResult 消息按 assistant 源顺序发出。
