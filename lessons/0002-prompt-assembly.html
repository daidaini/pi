提示词的动态拼装机制：系统约束与用户模板

本节目标：理解 pi 中 System Prompt 如何多模块拼接、Project Context 怎么融入，以及用户 `/` 命令如何转成 LLM 提示词。

本节完成后你应该能：

  • 解释 CWD、Date、Project Context 在 System Prompt 中起什么作用。

1. 系统提示词：八重拼装

打开 packages/coding-agent/src/core/system-prompt.ts 的 buildSystemPrompt 函数，你会发现它的主要职责就是组装字符串。在没有启用 customPrompt（即正常使用 pi 预设）时，它的拼装链条非常工整：

  1. 助手基本人设：固定前导句，说明自己是在 pi 里面的专家。
  2. 工具列表：基于 selectedTools，将工具名字和 toolSnippets 里的简短一句话介绍组合。注意：如果工具没有在 toolSnippets 中注册一句话介绍，它就不会出现在 Available tools 列表中（但仍可调用）。
  3. 操作指引（Guidelines）：组合默认规范和 promptGuidelines，去重。例如：Be concise、Show file paths clearly 等。
  4. pi 自家开发文档路径：在末尾注入 README.md、docs/、examples/ 的绝对路径，并重点交代“遇到 pi 本身开发任务时去这里读”。这是 pi 框架开发如此智能的核心秘诀之一！
  5. 自定义追加内容：拼入 appendSystemPrompt。
  6. 项目特定上下文（Project Context）：如果工作目录中有 AGENTS.md 等文件，会读出来，包在 <project_context> 标签里送给 LLM，实现“一项目一规则”。
  7. Skills 块（技能集）：如果可用工具有 read 且加载了 Skills，格式化后包在 <skill> 标签里追加进去。
  8. 当前时空信标：最底部加入 Current date 和当前绝对 Current working directory。由于模型不知道当前现实时间和目录，这一步对于它在执行 `write`/`edit` 相对路径时极其重要。

2. 源码导览：核心函数剖析

A. 标签式多文件包装

在 buildSystemPrompt 中，项目级 AGENTS 规则是以如下结构加入提示词的：

// Append project context files
if (contextFiles.length > 0) {
	prompt += "\n\n<project_context>\n\n";
	prompt += "Project-specific instructions and guidelines:\n\n";
	for (const { path: filePath, content } of contextFiles) {
		prompt += `<project_instructions path="${filePath}">\n${content}\n</project_instructions>\n\n`;
	}
	prompt += "</project_context>\n";
}

这种 XML 标签格式能最大程度减少模型定位规则的偏移，非常契合 Claude 等模型的长上下文指令遵循特性。

B. 用户模板展开 (Prompt Template)

当用户在会话框中输入 /foo arg1 "arg 2"，这并不是一个斜杠命令，而是模板展开！

打开 packages/coding-agent/src/core/prompt-templates.ts 的 expandPromptTemplate：

  1. 如果文本以 / 开头，通过正则 /^\/([^\s]+)(?:\s+([\s\S]*))?$/ 提取模板名 foo 和参数串 arg1 "arg 2"。
  2. 去 templates 数组（由 global / project prompts 扫描而来）里寻找名字匹配 foo 的模板文件（如 foo.md）。
  3. 通过 parseCommandArgs 拆分参数。这个解析器非常讲究，它用 inQuote 变量实现了对双引号/单引号括起来的参数不因空格截断的支持（例如 "arg 2" 仍被当作一个参数）。
  4. 调用 substituteArgs，通过正则批量替换骨架内容：
    • $1 替换成第一个参数 arg1
    • ${2:-default} 替换成第二个参数，若不存在则使用 default 默认值
    • $@ 或 $ARGUMENTS 替换成全量参数
    • 支持 bash 样式的参数切片 ${@:2}（代表从第 2 个参数开始往后的所有参数）。

我们在第一节学过，AgentSession.prompt() 是用户输入的入口。在 prompt() 执行时，它在什么位置调用这些展开逻辑？

看看 AgentSession.prompt() 里的时序 
// 1. 如果是 extension 命令行，立即在本地消费执行（不经过 LLM）
if (expandPromptTemplates && text.startsWith("/")) {
	const handled = await this._tryExecuteExtensionCommand(text);
	if (handled) return;
}

// 2. 触发 extension "input" 事件，允许 extension 拦截、改写 text 或 images
if (this._extensionRunner.hasHandlers("input")) { ... }

// 3. 此时才真正做展开 (Skill 展开 & Template 展开)
let expandedText = currentText;
if (expandPromptTemplates) {
	expandedText = this._expandSkillCommand(expandedText);
	expandedText = expandPromptTemplate(expandedText, [...this.promptTemplates]);
}

// 4. 发送给 Agent 之前的 before_agent_start 事件，允许 extension 修改最终 systemPrompt
const result = await this._extensionRunner.emitBeforeAgentStart(expandedText, ...);
