工具执行主调用链：模型决定，本地执行

这节只学一件事：当你在 pi 里看到 bash、read、edit 等工具执行时，源码里的主路径是什么。

本节完成后你应该能：
  • 画出“用户输入 → 模型产出 toolCall → 本地执行 → toolResult 回到模型”的链路。
  • 说清楚模型和本地 agent loop 的职责边界。
  • 知道下一次写架构文档时应先打开哪几个文件。

1. 先抓住边界

模型决定要不要调用工具。本地代码不会在用户 prompt 后直接挑一个工具运行；它把 active tools 的 schema 放进 LLM 请求，然后等待 provider stream 产出 ToolCall。AI 层的类型把这个边界写得很清楚：Context 里有 tools?: Tool[]，assistant 内容里可以出现 ToolCall，执行结果再以 ToolResultMessage 回到消息历史（见 packages/ai/src/types.ts）。

一句话：Tool 是“告诉模型能调用什么”；AgentTool.execute() 才是“本地真的怎么运行”。core agent 的 AgentTool 继承 AI 层 Tool，并增加 label、execute、executionMode（见 packages/agent/src/types.ts）。

3. 源码导览：按这个顺序读

A. 工具从哪里来？

内置工具入口在 packages/coding-agent/src/core/tools/index.ts。这里能看到 read、bash、edit、write、grep、find、ls 的 definitions 和 executable tools；createAllToolDefinitions() 是 product 层构建内置工具定义的入口（见 source）。

AgentSession._buildRuntime() 会创建这些 base tool definitions，再加载 extensions，并刷新 registry（见 source）。_refreshToolRegistry() 把 built-in、extension、SDK custom tools 合并，最后 setActiveToolsByName() 把可用工具写进 agent.state.tools（见 registry 和 active tools）。

B. 用户输入如何进入 loop？

AgentSession.prompt() 先处理 slash command、input extension、模板、鉴权、图片、before_agent_start 等 product 行为，然后调用 _runAgentPrompt()；后者调用 this.agent.prompt(messages)（见 _runAgentPrompt 和 prompt）。

C. 工具 schema 何时给模型？

在 core loop 的 streamAssistantResponse() 中，代码把 agent messages 转成 LLM messages，然后构造 llmContext：其中 tools: context.tools。这一步是“本地把工具选择权交给模型”的具体位置（见 source）。

D. 什么时候真正执行？

不是 toolcall_delta 一出现就执行。loop 等 assistant message 完成后，才过滤 message.content 里的 toolCall，调用 executeToolCalls()；执行结果再追加到 currentContext.messages（见 source）。

单个工具执行的核心顺序在 prepareToolCall() 和 executePreparedToolCall()：找工具、prepareArguments、validateToolArguments、beforeToolCall、tool.execute()、partial updates、错误转成 tool result（见 source）。执行后 afterToolCall 可修改结果，然后发 tool_execution_end，创建 ToolResultMessage（见 source）。

E. UI 为什么能显示“正在执行”？

interactive mode 订阅 AgentSessionEvent。当 assistant streaming update 中出现 toolCall 时，TUI 可以提前创建 pending 的 ToolExecutionComponent；当 tool_execution_start/update/end 到来时，再标记开始、刷新 partial result、写入最终 result（见 interactive-mode.ts）。单个工具行的状态和渲染在 tool-execution.ts。

4. 你现在应该记住的最小架构

AgentSession: 组装工具、选 active tools、处理产品层事件
      ↓
Agent / agent-loop: 把 Context.tools 交给模型，接收 toolCall
      ↓
Tool execution: validate → before hook → execute → after hook
      ↓
Events: tool_execution_start/update/end 给 UI / RPC / extensions
      ↓
ToolResultMessage: 追加到上下文，模型下一轮读取结果

不看上面的图，填这条链路中最关键的 3 个中间名词：

用户 prompt → → → → toolResult → 下一轮模型

目标答案：AgentSession、agent-loop、tool.execute。大小写不严格；你也可以写语义等价词。

如果要写架构文档，这节可直接转成哪段？

可写成“工具执行总览”：pi 在 session 层收集并激活工具，把工具 schema 传给模型。模型返回 assistant message，其中可能包含 tool calls。core agent loop 在 assistant message 完成后执行这些 tool calls，校验参数并经过 extension hooks，调用对应 AgentTool.execute()。执行结果通过事件给 UI/RPC，并作为 ToolResultMessage 追加进上下文，驱动下一轮模型响应。

常见错点自查
  • 如果你说“AgentSession 选择调用 bash”，不准确。AgentSession 只激活可用工具；模型输出 toolCall。
  • 如果你说“toolcall_delta 到了就执行”，不准确。执行发生在 assistant message 完成后。
  • 如果你忽略 ToolResultMessage，调用链就断了。工具结果必须回到下一轮模型上下文。
