# 耐用性 AgentHarness 与会话设计

<!-- 同步自 jot zmnps2zu。后续请在代码库中编辑此文件。 -->

耐用性（Durable）`AgentHarness` / 会话设计笔记。

## 框架设定

完全耐用性的 `AgentHarness` 本身是不现实的，因为重要的依赖项是由宿主应用提供的运行时 JS：

- 工具实现
- 模型/认证提供者
- 扩展和钩子处理器
- 资源加载器
- 系统提示词回调/修改器

工具注册表是运行时依赖。Harness 应当持久化可序列化的工具配置（例如活动工具名称），但不应持久化具体的工具实现。

实际的目标是半耐用性（semi-durable）Harness：

- 会话是耐用的、仅追加的状态树。
- Harness 将其拥有的状态持久化到会话条目中。
- 宿主应用负责在恢复时重新创建兼容的、不可持久化的依赖项。
- 恢复从耐用边界开始，而不是从正在进行的提供者流开始。

## 会话拥有耐用状态

将会话视为所有耐用的 Agent 状态，而不仅仅是对话记录历史。

现有的会话状态已经包含了 Harness 状态：

- 模型更改
- 思考层级更改
- 活动工具更改
- 叶节点条目
- 标签
- 压缩和分支摘要
- 自定义消息和自定义条目

这表明应继续使用单一的耐用会话日志，而不是添加 Harness 侧面文件（sidecars）。侧面文件对于大对象可能仍然有用，但会话条目应保持为“事实来源”的引用。

## 恢复时应用必须提供的项

应用必须重新创建兼容的运行时依赖项：

- 模型注册表 / 模型对象
- 工具注册表
- 扩展集、版本和顺序
- 资源加载器
- 系统提示词提供者/钩子
- 认证提供者
- 特定于应用的钩子

Harness 可以在可用时验证稳定的 ID/版本/哈希，但它自身无法序列化这些依赖项。

## 运行时配置与还原

构造函数选项保持为显式的运行时配置，不读取会话状态。在构造函数中进行隐藏的异步还原会使失败处理变得模糊。

未来的异步构建器/工厂应负责耐用性还原：

```ts
const harness = await AgentHarness.builder()
  .env(env)
  .session(session)
  .model(defaultModel)
  .tools(runtimeTools)
  .defaultActiveTools(["read", "edit"])
  .restore({ missingActiveTools: "fail" });
```

`restore()` 应读取活动分支，归约耐用性 Harness 配置，为缺失项应用默认值，针对应用提供的运行时依赖项进行验证，构造 Harness，并可选地在构造后发出 `source: "restore"` 更新事件。

对于活动工具：

- `active_tools_change` 条目是分支作用域的耐用配置。
- 如果分支上不存在 `active_tools_change`，还原将使用构建器默认值，或者如果未提供默认活动名称，则使用所有已注册的工具。
- 活动工具名称必须唯一。
- 工具注册表名称必须唯一。
- 默认情况下，恢复时缺失的活动工具名称应导致还原失败；稍后可以显式添加允许丢弃/禁用的宽松策略。
- 具体工具永远不会从会话中还原；宿主应用必须提供兼容的工具。

## Harness 应当持久化的内容

最小有用的耐用性条目：

- 分支作用域的活动工具名称
- 排队的 steer/followUp/nextTurn 消息
- 与轮次绑定的队列消耗
- 在活动操作期间接受的待处理会话写入
- 待处理写入的应用状态
- 操作的开始/结束/中断
- 轮次的开始/结束
- 提供者请求的开始/结束（如果恢复诊断需要）
- 工具调用的开始/结束（如果我们想要安全的工具恢复）

潜在的条目类型：

```ts
type DurableHarnessEntry =
  | QueueEnqueuedEntry          // 队列入队条目
  | QueueConsumedEntry          // 队列已消费条目
  | PendingWriteEnqueuedEntry   // 待处理写入入队条目
  | PendingWriteAppliedEntry    // 待处理写入已应用条目
  | OperationStartedEntry       // 操作已开始条目
  | OperationFinishedEntry      // 操作已结束条目
  | OperationInterruptedEntry   // 操作已中断条目
  | TurnStartedEntry            // 轮次已开始条目
  | TurnFinishedEntry           // 轮次已结束条目
  | ProviderRequestStartedEntry // 提供者请求已开始条目
  | ProviderRequestFinishedEntry // 提供者请求已结束条目
  | ToolCallStartedEntry        // 工具调用已开始条目
  | ToolCallFinishedEntry;      // 工具调用已结束条目
```

在公共 API resolve 之前，每个接受的变更必须已持久化。

## 恢复模型

启动时：

1. 宿主应用注册工具/模型/扩展/资源/认证/钩子。
2. Harness 打开会话。
3. Harness 将会话条目归约为：
   - 当前叶节点
   - 对话分支
   - Harness 配置（包括活动工具名称）
   - 队列
   - 待处理写入
   - 活动的操作/轮次/工具状态
4. Harness 验证所需的运行时依赖项，包括根据应用提供的工具注册表验证还原的活动工具名称。
5. Harness 调解（reconciles）未完成的操作状态。

提供者流不可恢复。恢复只能从耐用边界重试，或将操作标记为已中断。

## 恢复策略

默认的保守策略：

- 未完成的 Agent 轮次：标记为已中断，保留耐用队列/待处理写入，返回空闲状态。
- 未完成的提供者请求：标记为已中断；不自动重试。
- 未完成的工具调用：追加已中断/错误的工具结果；仅在工具声明为重试安全/幂等时才重试。
- 未完成的压缩：如果不存在压缩条目，则重新运行。
- 未完成的分支摘要/树导航：如果安全，则重新运行/应用缺失的摘要或叶节点条目。

可选策略：

```ts
recovery: "mark_interrupted" | "retry_unfinished"
```

对于非幂等的工具调用，必须谨慎使用 `retry_unfinished`。

## 关键场景

### 队列

- 在 `queue_enqueued` 之前崩溃：消息未被接受。
- 在 `queue_enqueued` 之后崩溃：消息已还原。
- 在队列消耗后但在耐用性轮次记录前崩溃：存在丢失/重复风险。
- 必要的等式：已消费的队列 ID 必须记录在 `turn_started` 或等效条目中，之后才被视为已消费。

### 待处理写入

- 在 `pending_write_enqueued` 之前崩溃：写入未被接受。
- 在入队后应用前崩溃：恢复过程将应用它。
- 在应用后标记前崩溃：确定的目标条目 ID 让恢复过程能够检测到条目已存在并将其标记为已应用。

### Agent 循环轮次

- 在提供者请求前崩溃：重试或标记为已中断。
- 在提供者请求期间崩溃：默认标记为已中断。
- 在提供者响应后助手消息持久化前崩溃：除非对提供者结果进行了日志记录（journaled），否则响应丢失。
- 在助手消息持久化后崩溃：从耐用消息中恢复。

### 工具调用

- 在工具调用开始后结果返回前崩溃：外部副作用可能已经发生。
- 默认恢复不应重新运行非幂等工具。
- 工具调用需要稳定的 ID 和重试安全性元数据以便自动恢复。

### 压缩

- 在摘要生成前崩溃：重新运行准备/摘要生成。
- 在摘要已生成但压缩条目记录前崩溃：除非摘要已记录日志，否则重新运行。
- 在压缩条目记录后崩溃：操作已完成；如果缺失则追加结束标记。

### 分支摘要 / 树导航

- 在摘要前崩溃：重新运行或标记为已中断。
- 在摘要条目后叶节点条目前崩溃：追加缺失的叶节点条目。
- 在叶节点条目后崩溃：操作已完成；如果缺失则追加结束标记。

## 最小可行性探索 (Minimum viable spike)

1. 添加耐用性队列条目。
2. 添加具有确定性目标 ID 的耐用性待处理写入条目。
3. 添加操作开始/结束/中断条目。
4. 添加带有已消费队列 ID 的轮次开始记录。
5. 通过归约会话日志进行恢复。
6. 默认将未完成的 Agent 轮次标记为已中断。
7. 仅在不存在最终条目时重新运行未完成的压缩/树操作。
8. 除非工具元数据说明重试安全，否则不重试未完成的工具调用。

- 应该为每个轮次对解析后的系统提示词文本进行快照以进行审计/调试吗？
- 恢复时是否要求严格的依赖项 ID/版本匹配？
- 应该记录多少提供者请求数据？
- 恢复过程是追加用户可见的助手中断消息，还是仅记录内部操作条目？
- 存储是否应支持在恢复期间截断最终的部分 JSONL 行？
